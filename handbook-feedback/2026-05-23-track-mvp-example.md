## 问题/建议

建议在**赛道介绍页**（如 Agentic Commerce、AI Security、Wallet/Permission、Governance 等）增加 **「4 周 Hackathon 最小可行原型」示例**。

### 理由

1. **降低方向选择门槛**：学员在 Week 2 初筛方向时，难以快速判断某个赛道的技术难度与自身能力是否匹配。一个具象的 4 周 MVP 示例（包含核心功能清单、技术栈建议、交付物定义）可以帮助学员做出更理性的决策。
2. **减少半途而废**：当前赛道介绍偏重概念和愿景，Builder 容易高估或低估实现成本。给出 MVP 边界后，学员能更早识别风险点并调整 scope。
3. **统一评审基准**：Hackathon 评委和 Mentor 也能基于类似的 MVP 定义给出更一致的反馈，避免「范围过大」或「demo 太浅」的反复拉锯。

### 建议格式

在每个赛道页面底部增加一个 `## 4 周 Hackathon MVP 示例` 小节，包含：

| 项目 | 说明 |
|------|------|
| **核心用户场景** | 1 句话描述谁在用、解决什么问题 |
| **最小功能集** | 3–5 个必须可演示的功能点 |
| **推荐技术栈** | 前端 / Agent / 合约 / 测试网 |
| **交付物清单** | README + 仓库 + 演示视频 + 合约地址 / 交易哈希 |
| **常见风险点** | 2–3 个往届 Builder 容易踩坑的地方 |
