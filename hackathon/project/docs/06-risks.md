# 风险边界分析

## 1. 服务端风险

### 1.1 重复扣款 (Double-Spend)

**风险描述**
Agent 因网络超时或重试逻辑错误，对同一请求发起多次支付。

**防御机制**
- x402 的 `idempotencyKey` 机制：服务端通过 facilitator 检查 key 是否已被使用
- Facilitator 在链上验证时拒绝已消费的 payment proof
- 服务端返回 `duplicate: true` 时直接返回之前的响应结果

**风险等级**: 中
**缓解后等级**: 低

### 1.2 支付验证绕过

**风险描述**
攻击者伪造 payment proof，跳过 facilitator 验证直接访问服务端。

**防御机制**
- 服务端必须将 proof 提交给 facilitator 验证，不可本地跳过
- 使用 HTTPS 防止中间人篡改 proof
- Facilitator 签名验证防止伪造结算证明

**风险等级**: 高
**缓解后等级**: 低

### 1.3 服务成本 > 收入

**风险描述**
AI 推理成本高于 $0.50/次，导致每次服务亏损。

**防御机制**
- 定价前测算真实推理成本（模型 API 费用 + 计算资源 + 带宽）
- 使用本地模型（Ollama/vLLM）降低成本
- 设置价格下限保护

**风险等级**: 高
**缓解后等级**: 中

### 1.4 Receipt 伪造

**风险描述**
攻击者伪造 `Payment-Receipt` header，欺骗 Agent 认为已付款。

**防御机制**
- Receipt 必须包含链上 txHash，Agent 可独立向网络验证
- Facilitator 签名防止伪造
- 不依赖服务端单方面声明

**风险等级**: 中
**缓解后等级**: 低

## 2. 客户端 (Agent) 风险

### 2.1 Pact 越权执行

**风险描述**
Agent 绕过 Pact 约束，向未授权合约或网络发起支付。

**防御机制**
- CAW 在服务端强制执行 Pact 检查，Agent 无法绕过
- Pact 规则由用户通过 CAW App 设置，Agent 只持有执行权限
- 每次支付前必须调用 `checkPact()`，拒绝在客户端跳过

**风险等级**: 高
**缓解后等级**: 低

### 2.2 预算耗尽攻击

**风险描述**
恶意服务方通过构造大量小额请求，快速耗尽 Pact 预算。

**防御机制**
- Pact 配置 `perTransactionMax`、`dailyLimit`、`totalTransactionsMax`
- 频率检查：短时间内过多请求触发告警或暂停
- 用户可手动冻结 Pact

**风险等级**: 中
**缓解后等级**: 低

### 2.3 Agent 被劫持

**风险描述**
Agent 运行环境被攻破，攻击者利用已授权的 Pact 发起恶意支付。

**防御机制**
- Agent 不持有私钥，支付签名由 CAW 安全模块完成
- Pact 的 `allowedContracts` 白名单限制攻击面
- `requireHumanConfirm: true` 时大额支付需人工确认
- 时间窗口过期自动失效

**风险等级**: 高
**缓解后等级**: 中

### 2.4 声誉欺骗

**风险描述**
服务方通过刷好评、伪造 ERC-8004 声誉记录欺骗 Agent。

**防御机制**
- 声誉评分需要多笔真实交易积累
- 链上记录不可篡改，历史争议记录永久可见
- Agent 应综合考量 completedJobs、disputedJobs、completionRate
- 新账户（"new" tier）应限制单笔金额

**风险等级**: 中
**缓解后等级**: 中

## 3. 链上风险

### 3.1 Gas 费波动

**风险描述**
Base Sepolia / 主网 Gas 费飙升，导致实际成本超出预算。

**防御机制**
- Pact 预算应预留 Gas buffer（如 $50 预算中 $2 为 Gas 备用）
- 使用 EIP-1559 动态 gas 估算
- 测试网阶段 Gas 可忽略，主网部署需重新测算

**风险等级**: 低
**缓解后等级**: 极低

### 3.2 网络拥堵

**风险描述**
支付交易pending时间过长，Agent 误判为失败并重试。

**防御机制**
- 设置合理的 timeout（如 30-60 秒）
- 重试时使用相同 idempotencyKey，防止重复扣款
- Facilitator 提供 tx status 查询接口

**风险等级**: 低
**缓解后等级**: 极低

### 3.3 智能合约漏洞

**风险描述**
Pact 合约或 ERC-8183 escrow 合约存在漏洞，资金被锁或被盗。

**防御机制**
- 使用经过审计的合约模板
- 测试网充分测试后再上主网
- 设置 `expiredAt` 超时自动退款机制
- 小额逐步验证，不一次性存入大额资金

**风险等级**: 中
**缓解后等级**: 低

## 4. 商业风险

### 4.1 交付物质量不达标

**风险描述**
Agent 已付款，但生成的内容质量差，不符合验收标准。

**防御机制**
- 引入 ERC-8183 Evaluator 机制：资金 escrow，验收通过后才释放
- 小额先试后付（pilot transaction），确认质量后再大额购买
- 分层验收：自动检查（违禁词/品牌词）+ 人工终审

**风险等级**: 高
**缓解后等级**: 中

### 4.2 争议无法仲裁

**风险描述**
"质量是否达标"存在主观分歧，没有公正的仲裁方。

**防御机制**
- 预先定义可量化的验收标准（如品牌词覆盖率 ≥ 80%）
- 引入第三方 Evaluator Agent（基于 ERC-8004 高声誉选择）
- 保留链上 deliverableHash，仲裁时比对原始交付物
- 超时自动退款作为无争议兜底

**风险等级**: 中
**缓解后等级**: 低

### 4.3 服务方跑路

**风险描述**
ContentGen Agent 收款后不交付，或永久下线。

**防御机制**
- Escrow 模式：资金先锁定，交付验收后释放
- 声誉系统：跑路行为永久降低声誉分，影响未来接单
- Pact 时间窗口：超时未交付自动触发 refund

**风险等级**: 中
**缓解后等级**: 低

## 6. 安全设计原则

1. **不信任客户端**：所有支付验证由服务端 + facilitator 完成，不依赖 Agent 自证
2. **最小权限**：Pact 只给必要预算和操作范围，不给无限授权
3. **可审计**：所有动作留痕，支持事后复盘和追责
4. **超时兜底**：任何未完成的交易都有自动退款或取消机制
5. **渐进验证**：小额试点 → 逐步放大，不一次性投入大额资金
6. **多层验收**：自动检查 + 人工审核 + 第三方仲裁，不依赖单一判断
