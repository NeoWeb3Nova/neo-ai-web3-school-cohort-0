### 2. Pact v2 检查逻辑（已运行验证）

从 `x402_client.py` 中提取的 v2 检查矩阵：

| 检查项 | 触发条件 | 拦截行为 | alert_level |
|--------|---------|---------|-------------|
| 预算 | 单次 > per_transaction_max | 拒绝 | blocked |
| 累计日限 | session_spent + price > daily_limit | 拒绝 | human_review |
| 小时频率 | hourly_tx_count > max_hourly | 拒绝 | human_review |
| 连续失败 | consecutive_failures >= 2 | 暂停 | blocked |
| 未知合约 | pay_to 不在白名单 | 拒绝 | human_review |
| approve 函数 | function_name == "approve" | 拒绝 | human_review |
| 临近过期 | expires < now + 2h | 警告 | human_review |
| 黑名单函数 | self_destruct / transfer_ownership / withdraw | 拒绝 | blocked |

## 关键决策

### 决策 1：Mock 优先，提交优先

- **背景**：6/4 尝试接入真实 x402 SDK 时，发现仅完成框架搭建，无法验证实际安装和测试网交互（缺少运行验证记录）
- **决策**：放弃 6/4 原计划中"真实链上交易验证"的执念，全力确保 6/12 提交时有一个**可运行的、可录制的、可解释的** Mock Demo
- **理由**：Hackathon 评审标准第 4 条是"可演示性"，第 5 条是"风险边界说明"——Mock 模式只要 README 中诚实标注边界，完全符合要求；而不可运行的"真实代码"反而违反第 4 条

### 决策 2：真实接入点保留，标注 TODO

- `x402_client.py` 中的 `CoboCAWWallet` 类已保留与真实 SDK 兼容的接口 (`check_pact`, `log_action`, `reset_session`)
- `x402_server.py` 中的 `generate_mock_content` 已独立为函数，未来替换为真实 AI 服务调用即可
- README 中增加"架构图：当前 Mock 组件 vs 未来真实组件"对照表

### 决策 3：威胁模型作为 Demo 核心亮点

- 8 种攻击场景已在 6/3 全部验证通过，v2 拦截率 87.5%
- Demo 视频的攻击演示段直接复用 `threat-model-simulator.py` 输出
- 这是 PactGuard 与其他 x402 Demo 的差异化亮点
