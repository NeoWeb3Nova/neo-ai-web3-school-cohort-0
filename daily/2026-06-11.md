## 笔记与思考

### 关键概念

#### 1. ERC-8004 Provider 元数据需要可交互的 UI 层
今天投入大量 commit 在 provider 的展示层：details modal、statistics overview、compact cards。这说明一个趋势：Agent 支付的供应商不再是静态配置，而是需要运营者查看、筛选、手动添加的动态白名单。provider 的元数据（name / version / chains / payment methods）必须对终端用户可见，而不仅是对 Agent 可见。

#### 2. Agent Card Assignment 是"人 → 卡 → Agent"的关键链路
新增 `agent card assignment flow`：用户在前端选择一张已审批的卡片，显式分配给某个 Agent。这解决了之前"卡片创建后自动关联 Agent"的隐式逻辑，改为显式授权，更符合最小权限原则。

#### 3. Real CAW Card Creation Loop Error 是 async 递归陷阱
修复了 real CAW client 创建卡片时的循环错误。原因：在 async 函数内部，如果某个条件判断后再次调用自身（递归），而事件循环没有正确 yield，会导致递归深度溢出或无限循环。FastAPI + async SDK 的组合下，这种错误很难在单元测试里捕获，因为 mock client 通常不模拟真实的异步网络延迟。

**解法**：在 real client 的创建路径里增加状态守卫（guard flag），确保同一卡片在 pending 状态下不会被重复提交创建请求。

### 问题与卡点

1. **手动 provider 表单展开后布局错位**
   - 原因：manual provider 表单使用了 `absolute` 定位，但父容器高度未预留
   - 解决：改用 `fold` 折叠模式，点击"添加自定义"才展开，正常态不占空间

2. **Dashboard 数据刷新时无 loading 状态**
   - 原因：用户会误以为页面卡死
   - 解决：新增 `live sync loading state`，用 Skeleton 组件在数据获取时占位
