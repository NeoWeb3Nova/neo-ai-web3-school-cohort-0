## 笔记与思考

### 关键概念

#### 1. 项目骨架的标准化意义
将 experiments 代码从 `experiments/x402-caw-agent-payment-loop/` 迁移到 `hackathon/project/` 不仅是路径变更，而是完成了以下转换：
- **从实验脚本 → 可交付项目**：`src/`、`tests/`、`docs/`、`demo/`、`submission/` 五目录结构符合开源项目惯例
- **从个人笔记 → 评审可读文档**：每份文档都有明确的读者（评委）和用途（打分）
- **从即兴代码 → 可复现工程**：`.env.example`、`.gitignore`、`requirements.txt`（待补）为外部复现做准备

#### 2. Open Day 启动会暴露的评审真实偏好
启动会主持人明确说了评委的浏览顺序：
```
第一眼 → Demo (视频/演示)
第二眼 → 项目信息 (README / Report / 看板)
```
这意味着：
- **Demo 视频是最高优先级的提交物**，没有视频等于主动放弃印象分
- README 必须在 30 秒内回答四个问题：解决什么问题 / 目标用户是谁 / 最小 MVP 是什么 / 团队角色分工
- 需要建一个**任务看板**（GitHub Projects），评委会看进度管理

#### 3. Cobo 赛道规则 × PactGuard 对照结论
通过 `08-rules-gap-analysis.md` 逐项检查，得到关键结论：
- ✅ 5 条硬约束中，4 条已满足；第 3 条（真实资金执行能力）缺测试网交易证据
- ✅ 5 个评审维度中，PactGuard 理论上全部命中
- 🔴 **最大 Gap**：测试网 Transaction Hash 和 Demo 视频
- 🟡 **次大 Gap**：`run_demo.py` 一键运行入口（评委 3 秒启动）

#### 4. "542 原则"与赛道选择
启动会透露：若某赛道报名过多，获奖概率会降低。PactGuard 所在赛道是 Cobo（01 Agent-Native Payments），目前不确定参与人数，但专注打磨比换赛道更有价值。

### 问题与卡点

1. **测试网交易证据（最大阻塞）**
   - 需要 Base Sepolia 钱包 + USDC 水龙头
   - x402 Python SDK 发布状态仍未知，可能需要继续用 Mock + 诚实标注
   - 行动计划：6/7 申请钱包，6/8 尝试跑通至少一次测试网转账

2. **Demo 视频尚未录制**
   - 脚本框架已在 6/5 确定，但尚未执行录制
   - 需要确保代码在录制时稳定运行
   - 行动计划：6/7 必须完成首次录制

3. **一键运行入口缺失**
   - 当前需要分别启动 server 和 client
   - 评委不会耐心读文档再手动启动两个进程
   - 行动计划：写一个 `run_demo.py`，自动启动 server（子进程）→ 等待 → 启动 client → 收集输出 → 自动关闭
