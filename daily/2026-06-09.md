## 笔记与思考

### 关键概念

#### 1. Pydantic `response_model` 是隐形的字段过滤器
后端 `mock_caw_client.py` 的 `_card_to_dict()` 返回了 12 个字段，但 `backend/models.py` 的 `CardResponse` 只声明了 7 个。FastAPI 的 `response_model` 会自动过滤未声明的字段，导致前端 `CardPact` 类型拿到的数据缺了 `agent_id`、`time_window`、`cooldown_hours` 等关键字段。

**教训**：前后端联调的第一步不是写 UI，而是对 JSON 做 diff。

#### 2. API Client 层是前后端的"翻译官"
新建 `web/src/api/caw.ts` 用 typed `fetch` 封装所有接口，把后端的 snake_case 字段映射到前端类型。这样做的好处：
- 组件层只导入一个 `cawApi` 对象，不直接写 `fetch`
- HTTP 错误统一处理，组件层只关心业务状态
- 未来切换后端地址或鉴权方式，只改这一处

#### 3. React 表单状态与 API 调用的时序陷阱
`handleApprove` 调完 API 后必须 `await loadCards()` 刷新列表，否则用户会看到旧状态。`actionLoading` 状态也要在 finally 里清理，否则按钮永远处于 disabled。

#### 4. Browser 自动化在 React 下的局限
`browser_click` 对 React 按钮的点击有时会因 refs 过期而失效。验证按钮触发 API 的可靠方式是：
1. 先 `browser_click` 尝试
2. 若状态未变，用 `browser_console` 执行 `fetch()` 直接调 API
3. 再用 `curl` 或 `browser_console` 读取结果确认

### 问题与卡点

1. **Approve 按钮点击后前端未刷新**
   - 原因：前端组件未在 `handleApprove` 成功后强制刷新列表
   - 解决：已确保 `await loadCards()` 在 API 调用后立即执行
   - 浏览器验证时因 ref 过期点击未生效，用 console fetch 绕过验证通过

2. **Backend 启动时默认进入 `real` 模式**
   - 原因：环境变量 `CAW_MODE` 未设置时，上次运行的 shell 可能保留了值
   - 解决：显式 `CAW_MODE=mock uvicorn main:app` 启动

### 今日收获

- **Pydantic response_model 是隐形过滤器**：如果模型字段少于实际返回的数据，FastAPI 会静默过滤。前后端联调的第一步永远是对 JSON 做 diff，而不是写 UI。
- **API Client 层是前后端的翻译官**：把 fetch、错误处理、类型转换封装在 `api/caw.ts` 里，组件层只关心业务逻辑，不关心 HTTP 细节。
- **React 状态刷新必须串行**：`await apiCall()` → `await refreshData()` → `setLoading(false)`，三步缺一不可。
- **Browser 自动化在 React 下有局限**：refs 过期后点击不生效，用 `browser_console` 执行 `fetch()` 是可靠的兜底验证手段。
