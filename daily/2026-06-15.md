## 笔记与思考

### 关键概念

#### 1. AI Security = 把模型放在隔离层里（摘自 Handbook）
AI × Web3 的安全不是「防止模型说错话」，而是防止模型错误、恶意上下文和工具滥用变成真实资产、权限或治理事故。

**与项目映射**：
- **Prompt Injection**：项目中 Agent 是否会读取外部合约 README 或网页内容？当前已做 Vendor 白名单校验，但未来接入多源信息时需分层标注上下文可信级别。
- **Tool Abuse**：EIP-712 policy 和 vendor 白名单已构成第一层防线，但还需增加 rate limit 和 budget 监控，防止高频小额支付耗尽预算。
- **Key Safety**：当前 mock 数据与真实测试网密钥分离，.gitignore 完整。长期应确保主私钥不进入任何模型上下文或日志。
- **Permission Isolation**：审批（approve）与支付（pay）已分离，但未来可增加「只读查询」「交易草稿生成」「自动执行」三级权限。
- **Audit Log**：当前有日志记录，但尚未做日志防篡改（如定期将关键日志 hash 锚定到链上）。

#### 2. Agent Identity = 从临时会话变成可验证的经济参与者（摘自 Handbook）
Agent Identity 不是给 Agent 起个名字，而是让用户、服务和其他 Agent 能验证：它是谁、谁控制它、能提供什么能力、服务入口在哪里、历史记录能不能追溯。

**与项目映射**：
- **Agent Profile**：「虚拟员工卡 Agent」可以有一份公开 profile，说明：名称、能力（创建卡片 / 审批 / 支付）、服务 endpoint、owner 钱包、版本号、隐私政策。
- **Capability**：应明确声明每个能力的风险等级（创建卡片=中风险，支付=高风险，审批=中风险）和输入/输出 schema。
- **Ownership**：当前项目由学员个人控制，长期应分离 operator（运行服务）与 owner（控制身份和关键更新），并用多签或 DAO 治理高价值操作。
- **Registry**：未来可将 Agent profile 注册到链上 registry，使其他服务或 Agent 能发现「虚拟员工卡」服务入口并验证其身份连续性。
- **DID / VC**：如通过第三方审计或能力测试，可签发可验证凭证（VC），增强外部信任。

### 今日收获

- **安全不是功能，是架构**：EIP-712 和 vendor 白名单是项目已有功能，但 AI Security 章节让我意识到需要在「输入层」增加 Prompt Injection 防护和「监控层」增加 rate limit / budget alert。
- **身份设计要前置**：如果项目未来进入 Agentic Commerce 或开放 API，必须有清晰的 Agent Profile、Capability 声明和 Registry 注册，否则其他 Agent 无法安全委托任务。
- **评审等待期是最适合做知识整理的阶段**：记忆新鲜、代码稳定、没有交付压力，把碎片笔记沉淀为个人知识手册，ROI 最高。
- **敏感信息扫描是持续动作，不是一次性检查**：即使提交前扫过，等待期代码变动后仍需重新确认，确保 public 仓库安全。
