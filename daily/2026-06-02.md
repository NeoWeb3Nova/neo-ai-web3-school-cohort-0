## 赛道详情对比

### Cobo 赛道｜Agentic Economy × Cobo Agentic Wallet

**核心关注**：Agentic Commerce —— 让 AI 参与经济活动的应用或基础设施。

**5 个建议方向**：

| 方向 | 一句话描述 | CAW 角色 |
|------|-----------|---------|
| 01 Agent-Native Payments | HTTP 402 自动完成支付，不依赖 API Key | Agent 持有和支配资金 |
| 02 Trustless Agent Work Agreements | ERC-8183 实现 Agent 间去信任工作协议 | Agent 资金账户，接收报酬、管理托管 |
| 03 Agent Resource Procurement | 自主发现、比价、采购算力/数据/API/存储 | 赋予 Agent 自主采购能力 |
| 04 Autonomous Trading | 自主执行链上交易策略（套利、做市、流动性挖矿） | 持有交易资金，授权边界内自主签署 |
| 05 A2A Economy | 多个 Agent 各自拥有钱包 → A2A 互雇、拍卖、分账、Treasury | 每个 Agent 独立 CAW 钱包 |

**赛道规则（硬性）**：
1. 项目必须围绕 Agent 与资金操作场景展开
2. Agent 的资金操作需通过 CAW 完成
3. Agent 需具备**真实的资金执行能力**（支付/转账/结算/收益管理/资产调度），**非纯流程设计**
4. 需体现 CAW 在钱包管理、权限控制、安全隔离、自主支付的价值
5. 最终成果需为**可运行/可演示的产品原型**，**不接受纯 PPT、概念设计或 Mockup**

**提交要求**：
- GitHub Repo
- README + 项目说明文档
- Demo 视频（建议 3–5 分钟）
- 项目演示链接（如有）
- 使用 CAW 的关键代码或配置说明
- 链上证据（如适用）：测试网地址、Transaction Hash、Agent Wallet 地址、流程截图

**评审侧重点**：
1. 场景贴合度：Agentic Commerce 叙事清晰度
2. CAW 关键性：CAW 是否是资金流程关键组件（非可替换展示元素）
3. 资金流程完整度：从任务触发到资金操作完成的主要流程
4. 可演示性：Demo 是否稳定展示核心流程
5. 风险边界说明：权限/账户/链上交互的基本边界和控制方式

### Z.AI 赛道｜Web3 × Long-Horizon Task

**核心关注**：GLM-5.1 的长程任务能力 —— 自主拆解复杂任务、制定多步骤计划、持续调用工具、迭代修复，完成从需求到交付的完整 Web3 工作流。

**3 个建议方向**：

| 方向 | 一句话描述 | 复杂度 |
|------|-----------|--------|
| 01 Agentic Dev Tools for Web3 | 一句话交付完整 Web3 工程（需求→设计→编码→测试→部署→文档） | 高 |
| 02 AI-Powered 3D World Builder | 从空间规划、场景生成、资产配置到可部署包的完整生产链路 | 中 |
| 03 AI × Creator Economy | 从创意输入、资料收集、创作、排版、发布到链上变现的完整内容生产链路 | 中 |

**赛道规则（硬性）**：
1. **推荐**使用 GLM-5.1 完成；Demo 中核心任务应由 GLM-5.1 驱动
2. 需围绕 Web3 × Long-Horizon Task，体现**长程、自主、持续执行**
3. Agent 需能自主拆解复杂任务、制定并执行多步骤计划、持续迭代/自我纠错
4. 完成一个有**实质复杂度**的 Web3 任务
5. 最终成果需为**可运行/可演示的产品原型**，不接受纯 PPT、API Mock

**提交要求**：
- GitHub Repo + README（说明目标、架构、运行方式、GLM-5.1 调用位置）
- 可运行 Demo 或演示链接 + 3–5 分钟视频
- **长程任务运行记录**：展示任务拆解、工具调用、迭代/修复过程、最终交付
- Web3 相关证明：合约地址、测试网交易、NFT/链上资产、钱包交互等
- 边界说明：安全边界、失败处理、人工介入条件

**评审侧重点**：
1. 赛道匹配度：是否围绕 Web3 × Long-Horizon Task（非普通问答/内容生成接口）
2. GLM-5.1 使用关键性：核心长程任务是否由 GLM-5.1 驱动
3. 任务复杂度与闭环程度：需求→计划→执行→验证→修复→交付完整链路
4. 长程稳定性：多步骤/多工具调用中是否保持目标一致
5. Web3 价值：是否解决 Web3 开发/3D设计/内容生产/链上资产/DAO协作的真实问题
6. 可演示性与可复现性
7. 安全、成本与权限边界

## 今日决策：赛道锁定

**继续锁定 Cobo 赛道。**

原因（基于今天获取的完整信息重新验证）：

1. **CAW 关键性验证**：PactGuard 的核心价值是 CAW Pact 作为支付前的硬性约束层——CAW 不是装饰，是签名前的最后一道网关。符合"CAW 关键性"评审标准。
2. **资金流程完整度**：从 Agent 发起请求 → x402 握手 → CAW Pact 预算/白名单/频率检查 → Session Key 条件签名 → 链上交易 → 审计日志，链路完整。
3. **真实资金执行能力**：虽然 Demo 用测试网，但代码架构支持真实 USDC 转账和 Session Key 授权，非纯流程设计。
4. **可演示性优势**：威胁模拟器可直接输出 v1 vs v2 拦截对比表格，攻击场景可视化强。
5. **与已有资产 100% 对齐**：Week 2 全部模块（Payment Flow / Agent Identity / Wallet Permission / Threat Model）自然融入。

**潜在风险（Cobo 赛道）**：
- "不接受纯 PPT、概念设计或 Mockup" → 必须在 6/12 前让代码真正跑起来，至少本地模拟端到端
- "真实的资金执行能力" → 如果无法上测试网，需要更强的本地链模拟 + 架构说明来证明"可迁移到真实环境"
