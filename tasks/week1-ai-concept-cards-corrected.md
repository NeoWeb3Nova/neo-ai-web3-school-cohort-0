## 卡片 1：Large Language Model (LLM)

### 你的原始表述

> 在训练过程中，模型在空间里建立**向量数据库**，通过向量之间的空间关系来体现并产生智能。

> 人工智能真正被点燃肯定是因为 **ChatGPT 3.5**

### 问题诊断

| 错误点 | 严重程度 | 说明 |
|--------|----------|------|
| "向量数据库" | 🔴 高 | 混淆了 LLM 内部的**词向量表示（embeddings）**与外部 RAG 系统的**向量数据库（vector DB）** |
| "ChatGPT 3.5" | 🟡 中 | 产品名与模型名混用。正式模型名是 **GPT-3.5**，产品名是 **ChatGPT** |
| "产生的智能化" | 🟢 低 | 口语循环定义，信息量低 |

### 修正表述

> LLM 在训练过程中学习的是 Token 的**高维向量表示（embeddings）**。这些向量在高维空间中的距离、方向关系编码了语义相似性和语法规则，使模型具备语言理解与生成能力。

> 生成式 AI 的大众爆发点通常被认为是 **OpenAI 于 2022 年底发布的 ChatGPT**（底层基于 GPT-3.5 模型）。

### 为什么错了

- **向量数据库**（如 Pinecone、Milvus、pgvector）是外部存储+检索系统，用于 RAG 场景。LLM 内部并没有一个可供查询的"数据库"，而是神经网络的权重矩阵中编码了向量关系。
- **ChatGPT 是产品名，GPT-3.5 是模型名**。就像"iPhone"和"A17 Pro 芯片"的关系，不能叫"iPhone A17"。

## 卡片 2：Prompt（提示词）

### 你的原始表述

> Prompt 已经可以通过**智能化生成**或 AI 辅助来完成格式化。

### 问题诊断

| 错误点 | 严重程度 | 说明 |
|--------|----------|------|
| "智能化生成" | 🟡 中 | 语义循环（用"智能"解释"智能"），没有说明具体机制 |

### 修正表述

> Prompt 的优化正在从**人工编写**走向**自动化优化**：
> - **自动提示优化**（Prompt Optimization）：如 DSPy、OPRO 等框架，用算法/小模型自动迭代生成更有效的提示词。
> - **系统提示词（System Prompt）模板化**：应用框架（如 LangChain、Agents SDK）在底层自动拼接系统指令，降低用户输入门槛。

### 为什么错了

- "智能化生成"是一个空泛的自我指涉词汇，没有传递任何具体机制。需要明确指出是**算法自动优化**还是**框架模板化**，才能把概念讲清楚。

## 卡片 3：Workflow（工作流）

### 你的原始表述

> 工作流就是通过逻辑链路和节点控制来接入大模型，并按照逻辑顺序来规范大模型的产出。

### 问题诊断

| 错误点 | 严重程度 | 说明 |
|--------|----------|------|
| 本节表述 | 🟢 低 | 基本正确，但可以更精确 |

### 修正表述

> Workflow 是**有向图（DAG）结构**的任务编排：节点代表操作（LLM 调用、工具执行、条件判断），边代表数据流和控制流。它把"一次对话"扩展为"多步骤、可分支、可回滚的业务流程"。

### 为什么需要修正

- 原始表述强调了"逻辑链路"但没有点出 Workflow 的核心特征：**图结构（Graph）**。如果只是线性链路，那叫 Pipeline；Workflow 的关键在于可以有分支（条件路由）、循环和并行节点。

## 卡片 4：Tool Use

### 你的原始表述

> 大模型在进行输入预测时，能够预测出它需要运行哪些工具，这种能力通常以"**钩子**"（Hooks）的形式出现。

> **MCP：这是一种通用的命令行接口协议**

> Skill 相比 MCP 有很多优点，但两者是相辅相成的，并不能完全互相取代。

### 问题诊断

| 错误点 | 严重程度 | 说明 |
|--------|----------|------|
| "Hooks" | 🔴 高 | LLM 调用工具的标准术语是 **Function Calling / Tool Calling**，不是 Hooks |
| "MCP 是命令行接口协议" | 🔴 高 | **完全错误**。MCP 是 Model Context Protocol，不是 CLI 协议 |
| "Skill 相比 MCP" | 🟡 中 | **层级不对等**。MCP 是协议层，Skill 是应用层封装，不能比"优缺点" |

### 修正表述

> 大模型在推理过程中输出**结构化调用指令**（如 JSON schema），由外部系统解析并执行对应工具。这种机制称为 **Function Calling**（OpenAI 术语）或 **Tool Calling**（通用术语）。

> **MCP（Model Context Protocol）**：由 Anthropic 于 2024 年提出的**开放协议**，标准化 AI 模型与外部工具/数据源之间的连接方式。类比：MCP 是"AI 的 USB-C"，统一接口标准，不关心具体传输什么数据。

> **MCP vs Skill（或 Tool）的关系**：
> - MCP 在**协议层**（protocol layer），解决"怎么连"的问题。
> - Skill/Tool 在**应用层**（application layer），解决"做什么"的问题。
> - 两者不是竞争关系，而是**分层协作**：Skill 可以通过 MCP 协议暴露给模型。

### 为什么错了

- **Hooks** 是编程范式中的回调机制（React hooks、Git hooks、系统钩子），和 LLM 输出结构化指令让外部执行工具是完全不同的概念。
- **MCP** 全称 Model Context Protocol，核心目的是**标准化上下文交换**，与命令行（CLI）无关。把它说成"命令行接口协议"是把 MCP 和 Bash/CLI 工具调用混淆了。
- **Skill 和 MCP 不在同一抽象层级**，就像你不能比较"HTTP 协议"和"网页应用"哪个更好。MCP 可以承载 Skill 的通信，Skill 可以在 MCP 协议之上运行。

## 卡片 5：AI Coding / Vibe Coding

### 你的原始表述

> 其实 AI Coding 领域还有一个非常红的词叫 **Web Coding**，如果我没记错的话，应该是 Karpathy 首先提出的。

### 问题诊断

| 错误点 | 严重程度 | 说明 |
|--------|----------|------|
| "Web Coding" | 🔴 高 | 记错了。Karpathy 提出的是 **Vibe Coding**，不是 Web Coding |

### 修正表述

> **Vibe Coding**（氛围编程）由前特斯拉 AI 总监 Andrej Karpathy 于 2025 年初提出。核心含义是：开发者不再逐行手写代码，而是通过自然语言描述"感觉/氛围"（vibe），让 AI 生成完整代码，人只做方向把控和审美校验。

### 为什么错了

- "Web Coding"在 AI 语境下几乎不存在，属于**记忆偏差**。Karpathy 的原帖明确用的是 **Vibe Coding**，强调的是"凭直觉/氛围驱动开发"，而非"Web（网页）开发"。

## 卡片 6：Human in the Loop

### 你的原始表述

> 人类参与其中的作用就是"**确权"**

### 问题诊断

| 错误点 | 严重程度 | 说明 |
|--------|----------|------|
| "确权" | 🟡 中 | "确权"是法律/经济学术语（确认产权归属），此处用"把关"或"授权确认"更准确 |

### 修正表述

> 人类在循环中的核心作用是**把关（Sanity Check）与授权确认（Authorization）**：
> - **校验**：确认 AI 输出的事实正确性和逻辑合理性。
> - **授权**：对涉及资产、权限、不可逆操作的步骤进行最终确认或拒绝。
> - **纠偏**：当 AI 偏离任务目标时，提供修正信号。

### 为什么错了

- **确权**（如不动产确权、数据确权）是法律概念，指确认某项权利归属于谁。在 AI 交互场景中，人类做的不是"确认所有权"，而是"校验内容并授予执行权限"。用"授权确认"或"把关校验"更贴切。

## 学习反思（费曼纠错笔记）

**这次纠错的收获**：

1. **层级意识很重要**：协议（MCP）≠ 应用（Skill）≠ 接口（CLI）。混用层级会导致选型逻辑混乱。
2. **术语精确性**：Hooks ≠ Function Calling，确权 ≠ 授权。术语错用会在技术沟通中暴露不专业。
3. **命名空间要分清**：产品名（ChatGPT）、模型名（GPT-3.5）、公司名（OpenAI）三者不可混用。
4. **避免自我指涉定义**："智能化生成"是循环定义，必须落到具体机制（模板、算法、框架）。

**下一次整理概念卡片时的 checklist**：
- [ ] 这个术语是否有明确的官方定义？
- [ ] 它属于哪个抽象层级（协议/框架/应用/算法）？
- [ ] 是否有容易混淆的近似术语（如向量 DB vs embeddings）？
- [ ] 是否引用了人名/公司名/产品名？确认拼写和归属。
